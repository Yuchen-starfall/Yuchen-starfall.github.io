Snakemake

传统的shell 脚本开发的流程，其是输入为导向，如果其中某个步骤发生了问题，可能需要很多的事件去定位发生问题的某一个或多个步骤进行debug。

而snakemake 则是一种以输出为导向，向后回顾backward-looking 的方法，其工作流首先确定需要的输出文件类型，接下来选择适当地输入文件及软件以得到对应的输出。而这种以输出为导向的方法导致了我们可以在发生意外失败的地方重新冲头开始，同时所有的输入文件将会在工作流中各自独立执行

snakemake 的工作流可以简单概括为：

  1. 首先定义一些规则
  2. 设置需要的输出类型，snakemake 将会判断需要何种软件或流程以获得对应的输出类型

Concept

规则（Rule）

rule all 是工作流的入口规则，用于明确声明最终需要生成的所有目标文件。它是流程的核心部分，告诉 Snakemake：“这些文件是我最终想要的，请确保它们被生成！”如果没有 rule all，Snakemake 可能只会执行部分规则（取决于命令行参数），导致结果不完整。

  1. 定义目标文件
    在 rule all 的 input 中列出所有需要生成的最终文件（如报告、汇总结果、可视化图表等）。Snakemake 会解析这些文件的依赖关系，自动执行所有必要规则。
  2. 作为默认入口
    当直接运行 snakemake（不带参数）时，默认执行第一条规则。通常会将 rule all 放在 Snakefile 的开头，确保流程从生成所有目标文件开始。
  3. 不执行具体操作
    rule all 通常没有 shell、script 或 run 代码块，因为它仅用于定义目标，不执行实际的计算步骤。

之后的rule，每个数据处理步骤定义为一个 rule，包含：

  • input：输入文件列表
  • output：输出文件列表
  • shell/script/run：生成输出的命令（Shell命令、外部脚本或Python代码）
  • 可选参数（如线程数、内存占用等）

通配符（Wildcards）

使用 {sample} 等通配符实现通用规则，适配不同输入（如多样本处理）。 expand()与通配符的搭配使用可以很好的构建出所需处理的文件列表。注意expand这个函数身生成的是所有参数的笛卡尔积（全组合）。若需要按顺序一一对应（如样本与对照组的配对），可使用 zip

正则表达式+通配符

默认情况下，Snakemake 的通配符（如 {sample}）会匹配任意字符（包括数字、字母、符号等）。通过正则表达式，可以限制通配符的格式，例如：

  • 确保通配符只匹配数字（如样本编号 001、002）。
  • 强制通配符包含特定前缀或后缀（如 patient_ 开头）。

依赖自动解析

Snakemake 根据输入输出文件的依赖关系，自动构建有向任务依赖关系（DAG），确定任务执行顺序，无需手动指定流程。

并且可以使用dry-run,模拟运行，展示出运行的代码。

  • 验证流程逻辑是否正确。
  • 检查输入文件是否存在、依赖是否闭环。
  • 预览执行顺序和任务数量。

nohup snakemake -s scipt.smk --resources mem_mb=64000 --latency-wait 60 > snakemake.log &
